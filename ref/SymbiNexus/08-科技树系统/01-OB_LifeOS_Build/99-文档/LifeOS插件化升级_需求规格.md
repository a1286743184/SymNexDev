# LifeOS 插件化升级需求规格文档

> 版本：v2.0  
> 文档状态：终稿  
> 创建日期：2025-12  
> 最后更新：2025-01-25

---

## 文档概述

本文档描述将 Obsidian Vault 中的 LifeOS 系统从"散装形态"升级为"插件驱动形态"的产品需求。升级后，系统将具备可持续迭代、统一升级、配置迁移等能力，同时保持用户现有操作习惯不变。

---

## 第一章 背景与现状分析

### 1.1 当前系统形态

LifeOS 是一个构建在 Obsidian 知识库基础上的个人管理系统，涵盖任务管理和财务管理两大核心板块。当前实现采用"散装形态"，具体表现为：

**代码层面**：
- 任务渲染逻辑封装在 `TaskDashboardKit.js` 中
- 财务渲染逻辑封装在 `FinanceVizKit.js` 中
- 智能录入相关逻辑在 `SmartInputKit.js` 中
- 价格追踪逻辑在 `price_dashboard.js` 中
- 可视化依赖 `echarts.js`（存放在 vault 目录内）

**入口层面**：
- 所有看板通过 `dataviewjs` 代码块调用工具库
- 渲染依赖 Dataview 插件提供的 `dv` 全局对象和执行环境
- 配置分散在多个 JSON 文件中（如 `今日聚焦.config.json`、`财务看板.config.json`）

**数据层面**：
- 任务数据以 `- [ ] / - [x]` 格式存储在 Markdown 文件中
- 财务数据以 frontmatter 形式的单文件账单存储，扩展支持 CSV 和 Ledger 格式

### 1.2 散装形态的核心问题

| 问题维度 | 具体表现 | 影响 |
|---------|---------|------|
| **版本漂移** | 旧用户 vault 中散落多个入口脚本 | 行为不一致，bug 修复无法统一触达 |
| **更新困难** | 无法像插件一样统一升级 | 用户只能重装整合包从头来 |
| **依赖脆弱** | 依赖 Dataview 插件和 vault 内 JS 文件 | 任何一个环节失效导致整体不可用 |
| **配置分散** | 配置散落在多个 JSON 和 MD 文件中 | 迁移成本高，难以统一管理 |
| **无法商业化** | 缺乏可控的交付与更新机制 | 难以推广或产品化运营 |

### 1.3 升级的核心驱动力

1. **产品化需求**：希望将 LifeOS 作为可推广的个人管理系统交付
2. **可持续迭代**：需要建立统一的版本管理和更新机制
3. **用户体验保障**：存量用户需要平滑升级，不能破坏现有数据和使用习惯
4. **能力边界清晰**：任务、财务两大板块需要各自有明确的控制台和配置入口

## 第二章 产品目标与愿景

### 2.1 总体目标

构建一个 **Obsidian 插件（lifeos-engine）**，作为 LifeOS 系统的唯一运行时引擎和配置中枢。插件驱动所有任务和财务相关的渲染、配置、索引和性能优化能力，而用户仍可通过熟悉的 Markdown 看板和配置页进行交互。

### 2.2 核心设计原则

| 原则 | 说明 |
|-----|------|
| **能力收敛** | 所有业务逻辑集中在插件内，不再依赖散装 JS 文件 |
| **入口统一** | DataviewJS 代码块替换为插件 CodeBlock 渲染 |
| **配置唯一** | 插件 data.json 作为配置唯一来源，支持自动迁移旧配置 |
| **体验延续** | 用户仍通过 md 文件与系统交互，但内容由插件驱动 |
| **可更新性** | 具备离线更新能力，保留用户配置和缓存 |

### 2.3 验收标准

升级完成后，系统需满足以下验收条件：

- ✅ 任务看板数据与旧版完全一致
- ✅ 财务看板统计与旧版完全一致
- ✅ 无需安装 Dataview 插件即可正常使用
- ✅ 无需 vault 内散装 JS 文件（echarts.js 等）
- ✅ 插件更新不覆盖用户配置（data.json 保留）
- ✅ 任务与财务各有一个控制台作为统一入口
- ✅ 控制台支持面板/弹窗快捷打开，也支持 md 壳渲染
- ✅ 拖动 md 看板位置后能按文件名重新绑定

## 第三章 功能需求规格

### 3.1 插件架构需求

#### 3.1.1 插件结构

```
.obsidian/plugins/lifeos-engine/
├── manifest.json          # 插件清单（版本、ID、依赖）
├── main.js                # 主入口（包含 Task/Finance 双引擎）
├── styles.css             # 样式文件
├── assets/
│   └── echarts.min.js     # 内置图表库
└── data.json              # 用户配置存储（保留）
```

#### 3.1.2 插件 ID 与命名

- **插件 ID**：`lifeos-engine`
- **显示名称**：`LifeOS Engine`
- **版本号**：从 `1.0.0` 开始

### 3.2 配置管理需求

#### 3.2.1 配置界面结构

插件设置界面分为两个 Tab：

| Tab 名称 | 功能 | 界面风格 |
|---------|------|---------|
| **任务** | 任务系统配置入口 | 与现有"今日聚焦配置.md"视觉一致 |
| **财务** | 财务系统配置入口 | 与现有"财务看板配置.md"视觉一致 |

#### 3.2.2 配置存储策略

- **唯一来源**：插件 `data.json` 作为所有配置的运行时来源
- **自动迁移**：首次启动时自动导入现有 JSON 配置文件
- **兼容导出**：可选导出功能，将配置同步写入 vault 的 JSON 文件供备份和查看
- **配置保留**：插件更新时不覆盖 `data.json`

#### 3.2.3 任务配置内容

与现有"今日聚焦配置.md"保持一致，包含：

- **来源配置（Sources）**：扫描目录列表、文件过滤规则
- **展示配置（Views）**：今日/逾期/未来天数阈值、排序方式、胶囊显示规则
- **性能配置（Performance）**：索引缓存开关、缓存文件名、最大扫描文件数

#### 3.2.4 财务配置内容

与现有"财务看板.config.json"保持一致，包含完整的 **6 分区配置体系**：

| 分区 | 功能 |
|-----|------|
| **Sources** | 支持目录/文件/CSV 三种来源，支持排除规则 |
| **Schema** | 字段名映射、日期推断开关、金额口径（type/正负）、CSV 基本口径、Ledger 正则入口 |
| **Normalization** | 渠道映射、分类映射、账户/支付工具映射 |
| **Views** | 月度/年度图表开关、明细表配置、默认排序、内容口径 |
| **Budgets & Goals** | 月预算、分类预算、预算占用显示 |
| **Performance** | 索引缓存开关、缓存文件名、最大扫描文件数 |

### 3.3 渲染引擎需求

#### 3.3.1 渲染入口替代

移除对 `dataviewjs` 的依赖，改为 Obsidian 原生的 Markdown CodeBlock 处理器：

| CodeBlock 类型 | 功能 | 示例 |
|---------------|------|------|
| `lifeos-task` | 渲染任务相关界面 | ` ```lifeos-task mode: today ``` ` |
| `lifeos-finance` | 渲染财务相关界面 | ` ```lifeos-finance mode: month period: 2026-01 ``` ` |
| `lifeos-config` | 渲染配置界面 | ` ```lifeos-config section: task ``` ` |

#### 3.3.2 md 壳文件策略

md 文件保留为"壳"，仅负责调用插件渲染：

```markdown
<!-- 任务控制台.md -->
\`\`\`lifeos-task
mode: console
\`\`\`

<!-- 月度财务看板.md -->
\`\`\`lifeos-finance
mode: month
period: 2026-01
\`\`\`
```

**关键特性**：
- 壳文件可自由移动到任意目录
- 插件通过文件名识别功能类型
- 路径变更后自动重新绑定

### 3.4 任务模块需求

#### 3.4.1 数据来源

与现有"今日聚焦.config.json"的 sources 概念一致：

- 扫描指定目录列表
- 支持文件通配符过滤（如 `*.md`）
- 支持排除规则（如 `excludePathIncludes`）

#### 3.4.2 任务解析规则

沿用现有解析口径，不改变业务逻辑：

| 格式 | 说明 |
|-----|------|
| `- [ ]` | 未完成任务 |
| `- [x]` | 已完成任务 |
| 日期标记 | 支持 `due:`, `repeat:`, `remind:` 等现有格式 |
| 标签/胶囊 | 识别现有标签语法和胶囊标记规则 |

#### 3.4.3 过滤与排序

复刻现有逻辑：
- 按日期分组：今日、逾期、未来 N 天
- 按状态筛选：完成/未完成
- 支持换周归档和迁移未完成逻辑

#### 3.4.4 性能要求

- 支持索引缓存，避免每次渲染全量扫描
- 可配置最大扫描文件数
- 支持增量刷新

### 3.5 财务模块需求

#### 3.5.1 数据格式兼容

保持对现有数据格式的完整兼容：

- **单文件账单**：frontmatter 形式的 Markdown 文件
- **CSV 导入**：支持 CSV 格式账单文件
- **Ledger 格式**：支持 Ledger 格式解析（可选入口）

#### 3.5.2 财务解析能力

延续现有的 6 分区配置能力：

- 从 frontmatter 提取字段（分类、金额、日期等）
- 支持从文件名推断日期
- 金额正负解释模式（type 字段或正负号）
- CSV 基本口径配置
- Ledger 格式正则入口

#### 3.5.3 渠道与分类映射

- 渠道名称映射（Normalization）
- 分类名称映射（Normalization）
- 账户/支付工具映射（Normalization）

#### 3.5.4 可视化能力

- **月度/年度看板**：饼图、热力图、趋势图、TopN
- **明细表**：可配置显示内容（子分类或原始输入）
- **预算显示**：月预算与分类预算占用进度
- **缓存复用**：索引缓存减少重复读文件

#### 3.5.5 ECharts 依赖

- **移除 vault 内依赖**：不再从 `08-科技树系统/.../echarts.js` 加载
- **内置图表库**：将 ECharts 作为插件资产打包
- **加载方式**：从插件目录注入或打包进 main.js

### 3.6 控制台（Console）需求

#### 3.6.1 控制台定位

控制台是任务/财务两大板块的**统一入口**，层级高于现有的"今日聚焦"和"月度财务看板"。

#### 3.6.2 财务控制台功能

| 功能区块 | 具体功能 |
|---------|---------|
| **快速操作** | 创建本月/指定年月月度看板、创建年度看板 |
| **配置入口** | 打开财务配置页、重建财务索引、清理缓存 |
| **总览摘要** | 本月支出/收入/结余/预算进度条 |
| **子系统入口** | 资产落盘子系统跳转（预留） |
| **诊断信息** | 账单源数量、解析失败计数、CSV/ledger 启用状态 |

#### 3.6.3 任务控制台功能

| 功能区块 | 具体功能 |
|---------|---------|
| **系统状态** | 今日聚焦统计、本周委托状态、索引健康度 |
| **快速操作** | 打开今日聚焦、打开任务配置、手动换周归档、重建任务索引 |
| **入口聚合** | 周委托列表、备忘、INBOX、开发需求等来源导航 |
| **诊断信息** | 扫描成功/失败计数、来源路径失效检测 |

#### 3.6.4 控制台打开方式

**方式一：Obsidian 面板/弹窗（推荐默认）**
- 通过命令面板调起：`打开任务控制台`、`打开财务控制台`
- 支持快捷键绑定
- 全屏面板或模态框展示

**方式二：md 壳文件（可选）**
- `任务控制台.md` / `财务控制台.md`
- 内容为 ` ```lifeos-task mode: console``` ` 或 ` ```lifeos-finance mode: console``` `
- 用户可自由拖动到任意目录
- 插件按文件名识别并自动更新路径绑定

### 3.7 创建看板能力

#### 3.7.1 插件命令

提供以下命令：

| 命令 | 功能 |
|-----|------|
| `创建月度财务看板` | 弹窗选择年月，自动生成 md 文件 |
| `创建年度财务看板` | 弹窗选择年份，自动生成 md 文件 |
| `重建任务索引` | 手动触发任务数据重新扫描 |
| `重建财务索引` | 手动触发财务数据重新扫描 |
| `打开任务配置` | 打开任务配置 Tab |
| `打开财务配置` | 打开财务配置 Tab |
| `打开任务控制台` | 打开任务控制台面板 |
| `打开财务控制台` | 打开财务控制台面板 |

#### 3.7.2 生成路径配置

- **默认路径**：可配置看板生成的目标目录
- **路径匹配**：插件记录文件名与路径的映射关系
- **重绑定机制**：用户拖动 md 文件后，路径失效时按文件名重新查找并更新配置

### 3.8 更新机制需求

#### 3.8.1 离线更新方案

不依赖 Obsidian 官方插件商店，通过离线脚本更新：

**脚本功能**：
- 用户输入 Obsidian Vault 路径
- 自动定位 `.obsidian/plugins/lifeos-engine/` 目录
- 备份旧版本到 `.backup/<timestamp>/`
- 白名单覆盖版本文件
- 跳过用户配置文件（data.json）

**文件覆盖策略**：

| 文件 | 行为 |
|-----|------|
| `manifest.json` | 覆盖 |
| `main.js` | 覆盖 |
| `styles.css` | 覆盖 |
| `assets/echarts.min.js` | 覆盖 |
| `data.json` | **保留**（用户配置） |
| `*.log` | 保留（可选） |

#### 3.8.2 更新脚本

| 平台 | 脚本 |
|-----|------|
| Windows | `update.bat` + `update.ps1` |
| macOS/Linux | `update.sh`（后续扩展） |

#### 3.8.3 安全要求

- 最小权限：只写插件目录，不操作其他位置
- 透明输出：打印将要写入的文件清单和保留的文件清单
- 备份机制：更新前自动打包备份
- 错误处理：失败时提示并可回滚

## 第四章 非功能性需求

### 4.1 性能需求

| 指标 | 要求 |
|-----|------|
| 首次渲染 | 在合理时间内完成扫描和渲染（具体时间待测） |
| 缓存复用 | 启用缓存后，重复渲染应显著快于首次 |
| 内存占用 | 不应导致 Obsidian 内存异常增长 |
| 最大扫描文件数 | 可配置上限，防止扫描过多文件 |

### 4.2 可靠性需求

| 需求 | 说明 |
|-----|------|
| 配置迁移 | 首次启动自动导入旧配置，失败时有明确提示 |
| 容错处理 | 源文件解析失败时记录日志，不影响整体渲染 |
| 回滚能力 | 更新失败时可恢复到上一版本 |
| 诊断入口 | 提供索引状态、源文件健康度等诊断信息 |

### 4.3 兼容性需求

| 维度 | 要求 |
|-----|------|
| Obsidian 版本 | 兼容当前 Obsidian 版本（以实际测试为准） |
| Dataview 插件 | 不再依赖，但已安装时旧 md 页面仍可兼容运行（可选） |
| 移动端 | 控制台和面板应支持移动端显示 |
| 主题 | 适配暗色/亮色主题，不依赖特定 CSS |

### 4.4 可扩展性需求

| 需求 | 说明 |
|-----|------|
| 模块化设计 | 任务/财务引擎独立，便于后续扩展新模块 |
| 配置导出 | 便于高级用户备份和迁移配置 |
| 商业化预留 | 支持功能开关和授权验证（未来扩展） |

## 第五章 迁移与兼容需求

### 5.1 配置迁移策略

| 步骤 | 操作 |
|-----|------|
| 1. 检测 | 扫描 vault 中是否存在旧配置文件 |
| 2. 导入 | 读取旧配置并写入插件 data.json |
| 3. 验证 | 验证配置完整性，缺失项使用默认值 |
| 4. 记录 | 记录迁移状态和来源版本 |

### 5.2 看板迁移策略

- 批量扫描库内所有使用 `dataviewjs` 的 md 文件
- 替换为对应的 `lifeos-task` / `lifeos-finance` / `lifeos-config` CodeBlock
- 保留原有参数配置，仅替换渲染入口

### 5.3 兼容性保障

- 迁移后原有功能口径保持一致
- 不改变业务逻辑和数据解释方式
- 如有调整，提供明确的变更日志

### 5.4 清理清单

迁移完成后，可安全删除的文件：

| 文件 | 说明 |
|-----|------|
| `08-科技树系统/.../TaskDashboardKit.js` | 已内置到插件 |
| `08-科技树系统/.../FinanceVizKit.js` | 已内置到插件 |
| `08-科技树系统/.../SmartInputKit.js` | 可选清理 |
| `08-科技树系统/.../echarts.js` | 已内置到插件 |
| 旧的 `*.config.json` | 已迁移到插件 data.json（可保留作为备份） |

## 第六章 用户体验规格

### 6.1 核心使用场景

#### 场景一：日常任务管理

1. 用户打开"今日聚焦"查看今日任务
2. 完成任务（编辑 md 文件）
3. 打开"任务控制台"查看本周状态
4. 执行换周归档操作

#### 场景二：日常财务管理

1. 用户打开"财务控制台"查看本月预算进度
2. 打开"25年01月财务看板"查看详细统计
3. 使用控制台按钮创建下月财务看板
4. 查看年度财务看板对比全年数据

#### 场景三：配置调整

1. 用户打开"财务配置" Tab
2. 调整渠道映射或分类映射
3. 保存后所有财务看板自动生效

### 6.2 界面规格

#### 控制台布局

```
+--------------------------------------------------+
| [标签页切换] 任务 / 财务                          |
+--------------------------------------------------+
|  [总览摘要区]                                     |
|  - 今日任务数 / 本月支出 / 预算进度               |
+--------------------------------------------------+
|  [快捷操作区]                                     |
|  [创建月度看板] [重建索引] [打开配置]             |
+--------------------------------------------------+
|  [入口导航区]                                     |
|  - 今日聚焦 | 周委托 | INBOX | 开发需求          |
+--------------------------------------------------+
|  [诊断信息]                                       |
|  - 源文件: 156 | 扫描成功: 155 | 失败: 1        |
+--------------------------------------------------+
```

#### 配置 Tab 布局

```
+--------------------------------------------------+
| [标签页切换] 任务 / 财务                          |
+--------------------------------------------------+
|  [来源配置]                           [编辑]      |
|  - 扫描目录: /任务/INBOX, /任务/今日...           |
|  - 排除规则: 不含 "归档" 的路径                   |
```
