## 七、用户痛点解决验证

### 7.1 痛点1: 2000+文档导致手机启动慢(4-6s) -> 新方案冷启动<2s

**当前瓶颈分析**:
- Obsidian启动时需加载全部插件(18个) + 索引全部文件(2000+) + 初始化Dataview
- LifeOS Engine的 `maxFiles:5000` 扫描上限说明文件量已接近性能边界
- 财务看板使用独立缓存文件(`财务看板.index.json`)说明实时扫描已不可接受

**新方案可行性**: 
- SQLite索引 + 懒加载Markdown内容: **可行**，冷启动<2s目标可实现
- 前提: 启动时仅加载SQLite索引（~1-2MB），不扫描Markdown文件
- 风险: 首次安装需全量索引2000+文件，可能需要10-30秒（与第一轮研究一致）

**结论**: 技术上可行，但需确保首次索引体验良好（进度条+后台执行）。

### 7.2 痛点2: 无通知权限/无小组件 -> 新方案有完整通知和小组件

**当前替代方案**:
- task-reminder: PowerShell脚本 + ntfy.sh推送（仅Windows端）
- 无Android端通知（Obsidian无此权限）
- 无桌面小组件

**新方案需求**:
- Android: WorkManager + flutter_local_notification / tauri-plugin-notification
- 桌面小组件: home_widget / tauri-plugin-widgets
- 需迁移task-reminder的全部配置（6时间段映射 + 提醒策略 + 写回功能）

**结论**: 无论选Flutter还是Tauri，通知和小组件都有成熟方案。关键是将task-reminder的6时间段映射逻辑完整迁移。

### 7.3 痛点3: 打开软件步骤过多 -> 新方案有桌面快速录入入口

**当前路径**: 解锁手机 -> 找Obsidian -> 打开 -> 等待加载(4-6s) -> 点击Command Orb -> 点击智能录入 = **6步+4-6s**

**目标路径**: 解锁手机 -> 点击桌面Widget -> 直接录入 = **2步+<1s**

**新方案可行性**:
- Android: home_widget支持直接打开应用特定页面并传递初始数据
- iOS: WidgetKit同理
- 关键: Widget需支持语音输入（MacroDroid Webhook的替代方案）

**结论**: 技术上可行。VoiceInputModule.js的MacroDroid集成需用原生语音输入替代。

## 八、核心宗旨验证

**宗旨**: "一个入口实现所有个人记录需求的整合"

**当前实现验证**:
- Smart Input Pro 是唯一入口: 11个分类模块覆盖所有记录类型
- autoClassify=true: 用户无需手动选择分类
- quickInput.autoSubmit=true: 极速录入路径
- other模块作为兜底: AI无法分类时自动归入INBOX

**新方案保障措施**:

| 保障项 | 要求 | 风险 |
|--------|------|------|
| 单一入口 | 新应用首页即录入界面 | 低 - UI设计可控 |
| 自动分类 | 保留11模块分类提示词 | 中 - 提示词需完整迁移 |
| 兜底机制 | other/INBOX降级路径 | 低 - 逻辑简单 |
| 快捷入口 | 桌面Widget直达录入 | 中 - 依赖平台能力 |

**结论**: 核心宗旨在新方案中可以得到保障，前提是AI分类管道完整迁移且快捷入口可用。

## 九、数据迁移风险评估

### 9.1 双写一致性

**当前状态**: 无双写机制。Markdown是唯一数据源，LifeOS Engine和Dataview均为只读索引。

**新方案双写场景**:
1. 新应用写入SQLite + Markdown: 需保证原子性
2. Syncthing同步Markdown到其他设备: 需处理冲突

**具体风险**:

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| SQLite与Markdown不一致 | 中 | 数据查询错误 | Markdown为真相源，SQLite仅作索引；定期校验 |
| Syncthing同步冲突 | 低 | 数据丢失 | 文件级冲突检测 + 手动合并 |
| 首次索引期间写入 | 低 | 数据遗漏 | 索引期间锁定写入队列 |

**建议方案**:
1. **MVP阶段**: Markdown优先写入，SQLite异步索引（最终一致性）
2. **正式版**: 写入时同时更新SQLite和Markdown（WAL模式保证原子性）
3. **校验机制**: 启动时比对SQLite记录数与Markdown文件数，不一致时触发重建

### 9.2 数据格式兼容性

**关键格式**:

| 数据类型 | 当前格式 | 迁移要求 |
|----------|----------|----------|
| 账单 | Markdown frontmatter (type/category/subcategory/amount/channel/date) | 新应用必须能读写相同frontmatter |
| 任务 | Tasks插件emoji格式 (📅/✅/❌) | 新应用必须兼容emoji标记 |
| 周委托 | 特定文件名模式 (周度委托列表YYYYWNN.md) | 新应用必须识别此命名规则 |
| 宠物档案 | 内联字段 (weight:: / date:: / birth::) | 新应用必须解析Dataview内联字段 |
| 价格记录 | 列表项内联字段 (price:: / sup:: / unit::) | 同上 |

**结论**: 新应用必须完整兼容Obsidian的frontmatter格式、Tasks插件emoji格式和Dataview内联字段格式，否则将导致数据迁移失败。
