### 1.2 关键发现(2025-2026最新)

**Flutter**:
- Flutter 3.41 将桌面支持正式脱离Beta，Windows/macOS/Linux成为一等公民
- 冷启动优化30%，复杂应用可达 800ms 以内
- `home_widget` 插件(v0.9.0)成熟支持Android/iOS桌面小组件
- `workmanager` 插件(v0.9.0+)支持Android后台任务调度
- `flutter_android_widgets` 可纯Dart定义Android小组件，无需XML/Kotlin
- 阿里/腾讯/字节等大厂核心产品深度应用

**Tauri 2.0**:
- Android/iOS支持已稳定，v2.10.0版本
- `tauri-plugin-widgets`(v0.3.0)支持Android/iOS/macOS原生小组件，Windows/Linux桌面小组件窗口
- `tauri-plugin-notification`支持全平台通知(含FCM/APNs推送)
- 包体积极小(2-5MB)，内存占用低(20-60MB)
- 从Electron迁移至Tauri 2.0的应用反馈"丝滑如原生"
- 但Rust后端开发门槛较高，生态相对年轻

**React Native**:
- RNW(React Native Windows)由微软维护，v0.82版本
- 基于UWP XAML，Windows支持可用但非核心场景
- Android原生能力需要原生桥接代码
- 生态丰富但Windows端支持弱于移动端

**Capacitor 8**:
- 2025年发布8.0版本，Android最低API 24
- 本质是WebView壳+原生插件桥接
- Windows支持依赖社区插件
- 性能受限于WebView渲染

**.NET MAUI 10**:
- 2025年11月随.NET 10发布，Android API 36.1绑定
- 原生支持Windows桌面和Android
- C#全栈能力，Blazor Hybrid集成
- 但跨平台UI一致性一般，社区规模较小

### 1.3 推荐方案: Flutter

**推荐理由:**

1. **Android原生能力完备**: `home_widget`+`workmanager`+`flutter_local_notification`三大插件成熟覆盖小组件/后台任务/通知需求
2. **Windows桌面成熟**: Flutter 3.41正式支持Windows，多窗口、系统托盘等API齐全
3. **冷启动达标**: 优化后可达800ms以内，满足<2s要求
4. **UI一致性高**: Skia/Impeller自绘引擎，双平台UI完全一致，记账看板/饼图/趋势图渲染质量有保障
5. **生态最强**: pub.dev 4万+包，大厂验证，问题排查资源丰富
6. **开发效率**: Hot Reload <200ms重载，迭代速度快

**不选Tauri的理由**: Rust后端开发门槛高，单人开发效率低；Android端生态不如Flutter成熟；WebView渲染性能对复杂看板(饼图/趋势图)不友好。

**不选RN的理由**: Windows端RNW维护力度下降；原生能力桥接代码量大；性能不如Flutter。

## 二、本地数据存储方案

### 2.1 当前数据结构分析

SymbiNexus当前数据结构:
```
SymbiNexus/
  01-经纬矩阵系统/    -- 任务/看板/备忘/规律事项/智能录入
  02-复盘系统/         -- 日复盘/周复盘
  03-成就系统/         -- 成就馆
  04-模板系统/         -- 各类模板
  05-生活坐标系统/     -- 角色档案/事件记录/美食/消费决策/生活经验
  06-财务系统/         -- 账单数据(按年/月分目录)/统计分析
  07-项目系统/         -- 公考等子项目
  08-科技树系统/       -- OB构建/公考学习
```

核心数据类型:
- **账单数据**: Markdown文件，含YAML frontmatter(type/category/subcategory/amount/channel/note等)
- **任务数据**: Markdown文件，含DataviewJS动态渲染
- **文档数据**: 2000+ Markdown文件，需全文检索
- **配置数据**: JSON配置文件

### 2.2 嵌入式数据库对比

| 维度 | SQLite + Room | ObjectBox | Realm |
|------|--------------|-----------|-------|
| **类型** | 关系型 | 对象型(含向量) | 对象型 |
| **读性能** | 中 | 高(2-10x SQLite) | 高(零拷贝) |
| **写性能** | 中 | 高(批量写入优化) | 中 |
| **FTS支持** | FTS5内置 | 无原生FTS | 无原生FTS |
| **向量搜索** | sqlite-vec扩展 | 内置向量数据库 | 无 |
| **跨平台** | 全平台 | Android/JVM/Go/iOS/Dart | 全平台 |
| **Flutter支持** | sqflite/drift | objectbox-dart | realm-dart |
| **数据迁移** | 需手写SQL | 自动迁移 | 自动迁移 |
| **包体积** | ~1MB | ~2MB | ~5MB |
| **成熟度** | 极高 | 高 | 高(MongoDB维护) |

### 2.3 推荐方案: SQLite + Room/drift 双层架构

**核心设计: 文件级兼容 + 数据库索引**

```
+----------------------------------+
|         应用层                    |
+----------------------------------+
|  SQLite/drift (索引+查询层)       |  <-- FTS5全文索引 + 结构化查询
+----------------------------------+
|  Markdown文件 (数据源层)          |  <-- Syncthing同步的实际文件
+----------------------------------+
```

**设计原则:**
1. **Markdown文件仍是数据真相源**(Source of Truth) -- 保持Syncthing兼容性
2. **SQLite作为索引和缓存层** -- 从Markdown的YAML frontmatter和正文提取结构化数据
3. **写入时双写** -- 先写Markdown文件，再更新SQLite索引
4. **启动时校验** -- 对比文件修改时间与索引时间戳，增量重建索引

**具体方案:**

```dart
// 数据库Schema设计
class SymbiNexusDatabase {
  // 文档索引表
  documents: {
    id: UUID,
    file_path: String,       // 相对路径
    title: String,
    module: String,          // 01-经纬矩阵/06-财务等
    sub_module: String,
    frontmatter: JSON,       // YAML解析后的完整frontmatter
    content_text: String,    // 纯文本内容(去Markdown语法)
    file_modified_at: DateTime,
    indexed_at: DateTime,
    file_hash: String,       // SHA256前8位，快速变更检测
  }

  // FTS5虚拟表
  documents_fts: FTS5(documents)  // 全文索引

  // 账单数据表(从Markdown提取的结构化数据)
  bills: {
    id: UUID,
    file_path: String,       // 关联源文件
    date: Date,
    type: String,            // 收入/支出
    amount: Double,
    category: String,
    subcategory: String,
    channel: String,
    note: String,
    create_date: DateTime,
    year: Int, month: Int, quarter: String, weekday: String
  }

  // 任务数据表
  tasks: {
    id: UUID,
    file_path: String,
    title: String,
    status: String,          // todo/doing/done
    priority: String,
    due_date: Date?,
    tags: List<String>,
    module: String,
  }

  // 同步状态表
  sync_status: {
    file_path: String,
    local_modified_at: DateTime,
    sync_state: String,      // synced/conflict/pending
    conflict_resolution: String?,
  }
}
```

### 2.4 全文搜索方案: SQLite FTS5

**选择理由:**
- SQLite FTS5在10万条文本中检索仅需10-20ms，2000+文档绰绰有余
- 零外部依赖，嵌入式部署
- 支持BM25排序、snippet高亮、中文分词(需配合jieba分词器)
- 与主数据库同一进程，无网络开销

**中文搜索增强:**
- 使用 `simple` tokenizer + 预分词方案：写入时用jieba分词后以空格连接存入FTS5
- 或使用 `unicode61` tokenizer 作为基础，配合应用层分词

**MeiliSearch不选理由**: 需要独立进程运行，移动端不适合；2000文档量级FTS5完全胜任。

### 2.5 Markdown文件兼容性保障

**核心策略: 文件格式不变，索引层可重建**

1. **读取**: 解析YAML frontmatter + Markdown正文，存入SQLite
2. **写入**: 按现有模板格式生成Markdown文件(保持命名规则`YYMMDD-序号-分类-子分类.md`)
3. **同步**: Syncthing同步文件后，应用检测文件变更触发增量索引
4. **冲突**: 保留Syncthing的`.sync-conflict`文件，应用层提供冲突合并UI

### 3.3 LLM路由策略

```dart
enum LLMProvider {
  glm, qwen, gemini, doubao,
  minimax, mimo, ali, longCat
}

class LLMRouter {
  // 优先级策略:
  // 1. 用户手动选择
  // 2. 根据任务类型自动选择(账单分类用GLM，长文本用LongCat)
  // 3. 根据网络状态选择(国内API优先，海外API备选)
  // 4. 根据成本选择(免费额度优先)
  // 5. 离线时降级到本地模型

  Future<LLMResponse> route(String input, PipelineStage stage) async {
    if (!hasNetwork) return localFallback(input, stage);
    final provider = selectProvider(stage);
    try {
      return await provider.complete(input, stage.prompt);
    } catch (e) {
      return await retryWithNextProvider(input, stage);
    }
  }
}
```

## 五、Syncthing集成方案

### 5.1 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A: 嵌入Syncthing核心** | 一体化体验，无需额外安装 | Syncthing非设计为库，Go模块耦合严重；需自行编译Go为SO；维护成本极高 |
| **B: 调用外部Syncthing App** | 成熟稳定，零维护 | 用户需安装两个App；跨App通信复杂 |
| **C: Syncthing-Fork包装** | 基于官方代码，可定制UI | 仍需独立进程；Go编译环境要求 |
| **D: 自建轻量同步引擎** | 完全可控，深度集成 | 开发成本高；需实现冲突检测/增量同步/断点续传 |

### 5.2 推荐方案: B(调用外部Syncthing) + 渐进增强

**Phase 1 (MVP): 依赖外部Syncthing App**
- Android: 检测Syncthing-Fork是否安装，未安装则引导安装
- Windows: 检测Syncthing进程是否运行，未运行则启动
- 通过Syncthing REST API(默认8384端口)监控同步状态
- 应用内嵌入Syncthing Web管理页面的简化视图

**Phase 2 (增强): 嵌入Syncthing二进制**
- 将Syncthing Go编译为Android JNI库(.so)
- 通过ProcessBuilder启动Syncthing子进程
- 参考BasicSync项目的实现方式
- Windows端直接内嵌syncthing.exe

**Phase 3 (可选): 自建同步引擎**
- 仅同步应用的SQLite索引文件 + Markdown文件
- 基于WebSocket的P2P通信
- 增量同步(基于文件hash差异)

### 5.3 同步状态监控

```dart
class SyncthingMonitor {
  // 通过REST API获取状态
  // GET http://localhost:8384/rest/system/status
  // GET http://localhost:8384/rest/db/completion?folder=symbinexus

  Stream<SyncStatus> monitorSync() async* {
    while (true) {
      final status = await _api.getCompletion();
      yield SyncStatus(
        state: status.state,          // idle/scanning/syncing/error
        completion: status.completion, // 0-100%
        lastSync: status.lastSync,
        conflicts: status.conflicts,
      );
      await Future.delayed(Duration(seconds: 5));
    }
  }
}
```

### 5.4 文件冲突处理策略

```
冲突检测: Syncthing生成 .sync-conflict-YYYYMMDD-HHMMSS-DEVICEID.md 文件

处理策略:
1. 自动合并(推荐):
   - 解析冲突文件和原文件的frontmatter
   - 如果冲突仅在frontmatter字段(如status变更)，自动合并
   - 如果正文内容冲突，保留两个版本，标记待人工处理

2. 用户选择:
   - 应用内显示冲突对比视图
   - 用户选择保留哪个版本，或手动合并

3. 时间戳优先:
   - 对于账单数据，以create_date为准
   - 对于任务数据，以最后修改时间为准
```

### 6.4 关键技术决策记录(TDR)

#### TDR-001: 选择Flutter而非Tauri

```yaml
decision:
  title: "跨平台框架选择: Flutter vs Tauri 2.0"
  context: "需要同时支持Android(含小组件/通知)和Windows桌面，单人开发"
  options:
    - name: "Flutter 3.41+"
      pros: ["Android原生能力插件成熟(home_widget/workmanager)",
             "Windows正式支持", "生态最丰富(4万+包)",
             "Hot Reload开发效率高", "Skia自绘引擎看板渲染质量好"]
      cons: ["包体积较大(8-15MB)", "Dart语言需学习",
             "桌面小组件仍需部分Kotlin代码"]
    - name: "Tauri 2.0"
      pros: ["包体积极小(2-5MB)", "内存占用低",
             "tauri-plugin-widgets支持小组件",
             "Rust后端性能极佳"]
      cons: ["Rust学习曲线陡峭，单人开发效率低",
             "WebView渲染对复杂看板不友好",
             "Android端生态不如Flutter成熟",
             "中文社区资源较少"]
  decision: "Flutter 3.41+"
  rationale: "Android原生能力是核心需求，Flutter的home_widget/workmanager插件远比Tauri成熟；单人开发效率是关键约束，Dart比Rust上手快得多；记账看板的饼图/趋势图在Skia渲染下质量远优于WebView"
  consequences: "包体积会偏大但可接受；桌面小组件需要少量Kotlin桥接代码；团队扩展时Dart开发者比Rust更容易招聘"
```

#### TDR-002: SQLite + Markdown双层存储

```yaml
decision:
  title: "数据存储策略: SQLite索引 + Markdown文件"
  context: "2000+ Markdown文件需与Syncthing同步，同时需要高效查询和全文搜索"
  options:
    - name: "纯Markdown(当前方式)"
      pros: ["Syncthing天然兼容", "Obsidian可继续使用"]
      cons: ["查询性能差", "全文搜索需遍历文件", "统计看板计算慢"]
    - name: "纯SQLite(迁移所有数据)"
      pros: ["查询性能极佳", "FTS5全文搜索", "统计计算快"]
      cons: ["丢失Syncthing文件级同步兼容性",
             "无法与Obsidian共存",
             "数据迁移风险大"]
    - name: "SQLite索引 + Markdown源文件"
      pros: ["保持Syncthing兼容", "查询性能好",
             "FTS5全文搜索", "可降级回Obsidian"]
      cons: ["双写一致性需保障", "索引重建逻辑复杂",
             "存储空间略增"]
  decision: "SQLite索引 + Markdown源文件"
  rationale: "Syncthing文件级同步是硬性需求，纯SQLite方案会破坏兼容性；双层架构兼顾了查询性能和同步兼容性；索引可从文件重建，不存在数据丢失风险"
  consequences: "需要实现可靠的索引同步服务；写入操作需双写保障；应用首次启动需建立索引(2000文件约10-30秒)"
```

#### TDR-003: Syncthing外部依赖

```yaml
decision:
  title: "Syncthing集成方式: 外部App依赖"
  context: "MVP阶段需快速验证，嵌入Syncthing开发成本极高"
  options:
    - name: "嵌入Syncthing Go库"
      pros: ["一体化体验"]
      cons: ["Syncthing非设计为库，Go模块耦合严重",
             "需维护Go编译环境", "开发成本极高"]
    - name: "外部Syncthing App + REST API"
      pros: ["零维护成本", "成熟稳定", "快速实现"]
      cons: ["用户需安装两个App", "跨App通信"]
  decision: "外部Syncthing App + REST API(MVP阶段)"
  rationale: "MVP阶段优先验证核心功能，Syncthing集成属于基础设施层；外部依赖方式可在1天内实现，嵌入方式需1-2周；后续可渐进增强为嵌入方式"
  consequences: "Android用户需安装Syncthing-Fork；需处理Syncthing未安装/未运行的边界情况；后续Phase 2可嵌入二进制"
```

### 6.5 风险登记

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| Flutter桌面小组件需Kotlin桥接代码 | 开发量增加 | 高 | 使用flutter_android_widgets减少原生代码；小组件功能精简 |
| SQLite索引与Markdown文件不一致 | 数据错误 | 中 | 文件哈希校验；启动时增量同步；写入时事务保障 |
| Syncthing冲突文件处理复杂 | 数据丢失 | 中 | 自动合并策略；冲突对比UI；定期备份 |
| LLM API不可用(网络/额度) | 功能降级 | 中 | 8个提供商轮换；本地规则引擎兜底；离线模式 |
| 冷启动索引重建耗时 | 用户体验差 | 低 | 增量索引；后台异步重建；启动页进度提示 |
| Flutter Windows桌面性能 | 卡顿 | 低 | Impeller渲染器；列表虚拟化；图片缓存 |
