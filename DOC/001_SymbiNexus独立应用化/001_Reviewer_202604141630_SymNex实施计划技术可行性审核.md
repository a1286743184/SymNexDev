### Critical Issues (Must Fix Before Merge)

#### 1. [Schema]: 数据库表定义严重缺失 -- 4个AI模块无对应表

- Location: Phase 1 / P1-02 数据库Schema定义
- Impact: **High**
- Explanation: 计划声称支持11个AI模块，但drift Schema仅定义了7张表(Bills, Tasks, Memos, Contacts, PriceRecords, PetGrowth, AiLogs)。以下4个模块在data.json中明确启用，却无对应表定义:

  | 缺失模块 | data.json中的key | 需要的表 | 数据特征 |
  |----------|-----------------|---------|---------|
  | 题目录入 | question_entry | question_entries | section, question_identifier, optimized_content, exam_name |
  | 代码开发 | code_dev | (复用Tasks表?) | task_title, task_context, due_date, time_info |
  | 美食收藏 | food_wishlist | food_wishlist | shop, status, cuisine, location, dishes(JSON) |
  | 刷题记录 | study_record | study_records | exam_name, section_name, time_spent, total_questions, correct_answers, wrong_answers |

  此外，`other`(杂项记录)模块也无对应表，当前设计将其写入INBOX.md但不入库，将导致无法在App内检索。

- Suggested Fix:
  - 新增 `QuestionEntries` 表、`FoodWishlist` 表、`StudyRecords` 表
  - `code_dev` 模块的输出格式(task_title, task_context, due_date, time_info)与Tasks表高度同构，建议复用Tasks表并增加 `sourceType = 'code_dev'`，但需确认字段映射
  - `other` 模块建议新增 `MiscRecords` 表或复用Memos表(sourceType='other')
  - 同步更新 `@DriftDatabase(tables: [...])` 注解

#### 2. [Schema]: AiLogs表缺少关键字段，无法支撑Token追踪需求

- Location: Phase 1 / P1-02 / ai_logs_table.dart
- Impact: **High**
- Explanation: AiLogs表定义了 `inputTokens`/`outputTokens`/`cost`，但P2-01c的TokenTracker需求包含"月度统计"功能，缺少以下字段:
  - `pipelineStage` (TextColumn): 区分classify/optimize/extract三阶段，否则无法统计哪个阶段消耗最多
  - `rawInputHash` (TextColumn): 去重统计需要，同一输入可能触发多次调用
  - `latencyMs` (IntColumn): 响应延迟追踪，对供应商选择有参考价值
  - `errorMessage` (TextColumn): 失败原因记录，status='failure'时必须有

- Suggested Fix: 在AiLogs表增加上述4个字段，并添加索引 `(module, timestamp)` 和 `(provider, timestamp)` 以加速月度统计查询

#### 3. [Security]: API密钥存储方案存在路径泄露风险

- Location: Phase 2 / P2-01a / LlmProviderConfig
- Impact: **High**
- Explanation: 现有data.json中API密钥通过 `apiKeyPath` 引用md文件(如 `08-科技树系统/01-OB_LifeOS_Build/03-密钥/GLM4_API_KEY.md`)。计划使用 `flutter_secure_storage` 存储密钥，但未说明:
  1. 迁移方案: 如何从md文件读取密钥并安全导入flutter_secure_storage
  2. 密钥路径是否会在UI或日志中暴露
  3. Windows平台flutter_secure_storage使用DPAPI，但未提及fallback策略

- Suggested Fix:
  - 增加P2-01a子任务: "API密钥安全迁移"，包含从md文件读取 -> 写入flutter_secure_storage -> 验证读取的完整流程
  - 确保apiKeyPath仅用于首次导入，不持久化到App数据库
  - Windows平台需测试flutter_secure_storage的DPAPI可用性

#### 4. [Platform]: Syncthing双通道方案在Android 11+存在根本性障碍

- Location: Phase 3 / P3-03 / SyncService
- Impact: **High**
- Explanation: 计划提出"文件系统监听(主) + Syncthing REST API(辅)"双通道方案，但在Android 11+上存在以下根本性问题:

  **文件系统监听通道:**
  - Android 11+的Scoped Storage限制: 应用无法直接访问 `/storage/emulated/0/` 下的任意目录
  - `watcher` 包底层使用 `FileSystemEntity.watch()`，在Android上依赖inotify，但SAF(Uri)路径不支持inotify
  - 即使通过SAF获得目录访问权限，返回的Uri无法直接被File类和watcher包使用，需要通过ContentResolver间接操作

  **Syncthing REST API通道:**
  - Syncthing-Fork在Android上默认仅监听127.0.0.1:8384
  - Android 11+对localhost回环访问无限制，但需要确认Syncthing-Fork的API是否启用认证
  - 如果Syncthing-Fork未运行，REST API不可用

- Suggested Fix:
  - **方案A(推荐)**: 放弃文件系统监听，改用SAF + 定时轮询。使用WorkManager每15分钟通过SAF查询最近修改的文件，与DB索引比对后增量更新
  - **方案B**: 引入 `saf_stream` 或 `android_storage` Flutter插件，封装SAF的ContentResolver查询，替代直接File操作
  - **方案C**: 将Syncthing同步目录配置到App私有目录(`getExternalFilesDir()`)，该目录不受Scoped Storage限制，但需用户手动配置Syncthing指向新路径
  - 无论哪种方案，都必须在P0阶段验证可行性，而非等到Phase 3

#### 5. [Platform]: WorkManager + flutter_local_notifications在Android 14+存在严格限制

- Location: Phase 3 / P3-01 / BackgroundWorker + NotificationScheduler
- Impact: **High**
- Explanation:
  - **WorkManager限制**: Android 14+对后台任务有更严格的限制，`PeriodicWorkRequest` 最小间隔为15分钟，无法实现"每15分钟检查即将到期备忘"的精确提醒
  - **精确闹钟限制**: Android 14+要求 `SCHEDULE_EXACT_ALARM` 权限需用户手动授权(设置->应用->特殊权限)，且Google Play政策仅允许闹钟/日历类应用使用
  - **flutter_local_notifications**: 在Android 14+上，`zonedSchedule()` 依赖 `AlarmManager.setExactAndAllowWhileIdle()`，需要上述权限
  - **Doze模式**: 即使获得权限，Doze模式下仍可能延迟通知

- Suggested Fix:
  - 使用 `AlarmManager` + `BroadcastReceiver` 替代WorkManager实现精确通知调度
  - 在首次设置通知时，引导用户授予 `SCHEDULE_EXACT_ALARM` 权限
  - 对于非精确通知(每日汇总)，继续使用WorkManager的15分钟间隔
  - 增加 `android.permission.SCHEDULE_EXACT_ALARM` 和 `android.permission.USE_EXACT_ALARM` 到AndroidManifest.xml
  - 在P5-01集成测试中，必须在Android 14+真机上验证通知准时率

#### 3. [Architecture]: 三阶段管道的Stage 2(语义优化)设计存在效率问题

- Location: Phase 2 / P2-01b / SmartInputPipeline
- Rationale: 当前设计是 classify -> optimize -> extract 串行三阶段，但:
  - data.json显示 `bill.enableOptimization: false`，`question_entry.enableOptimization: false`，`food_wishlist.enableOptimization: false`，`study_record.enableOptimization: false` -- 11个模块中4个禁用了优化，意味着优化阶段仅对7个模块生效
  - 优化阶段需要一次额外的LLM调用，增加延迟和成本
  - 更重要的是: extract阶段的提示词(如bill的extractionPrompt)已经包含了"混合表达解析"、"去噪"等优化逻辑，与optimize阶段功能重叠
  - 计划中optimize阶段的提示词仅一个通用版本，但不同模块的优化需求差异很大(账单需要金额解析，备忘需要时间提取)

- Suggestion:
  - 考虑将optimize和extract合并为单次LLM调用，在extract提示词中内嵌优化规则，减少一次API调用
  - 如果保持三阶段，需为每个模块编写独立的optimize提示词(而非一个通用版本)
  - 增加管道配置: 允许用户选择"快速模式(跳过优化)"和"精确模式(三阶段)"

#### 4. [Architecture]: 提示词管理方案不够灵活

- Location: Phase 2 / P2-01b / PromptRepository
- Rationale:
  - 提示词存放在 `assets/prompts/` 下，打包进APK后无法热更新
  - defaults.js中的提示词包含动态变量(如 `${petNames}`, `${currentDate}`, `${existingShops}`, `${sysDateStr}`, `${weekDayStr}`)，计划中未说明如何处理这些变量注入
  - 用户自定义提示词覆盖(PromptRepository.setUserPrompt)的存储位置未明确

- Suggestion:
  - 提示词模板使用Mustache/类似语法替代 `${var}`，避免与Dart字符串插值冲突
  - 默认提示词打包在assets中，用户自定义提示词存储在 `shared_preferences` 或SQLite
  - 动态变量在运行时注入: `petNames` 从Contacts/PetGrowth表查询，`currentDate` 从PivotService获取，`existingShops` 从FoodWishlist表查询
  - 考虑支持从远程加载提示词更新(可选)

#### 5. [Performance]: 冷启动<1.5s目标在Android低端设备上难以达成

- Location: Phase 4 / P4-05 / 性能优化
- Rationale:
  - drift数据库的 `NativeDatabase.createInBackground()` 在首次启动时需要: 加载sqlite3_native.so -> 创建数据库文件 -> 执行migration -> 创建FTS5虚拟表
  - Riverpod的 `@Riverpod(keepAlive: true)` Provider在启动时全部初始化
  - 文件监控服务(FileWatcherService)启动需要遍历监控目录
  - Flutter引擎本身在Android上的冷启动时间约300-800ms(取决于设备)
  - 综合以上，在骁龙6系/天玑7系等中端芯片上，冷启动1.5s非常紧张

- Suggestion:
  - 将冷启动目标修改为: 旗舰设备<1.5s，中端设备<2.5s
  - 实施延迟初始化: 数据库和文件监控在首页渲染完成后再初始化
  - 首页(TodayFocusPage)使用 `FutureBuilder` + 骨架屏，避免阻塞渲染
  - 考虑使用 `flutter_native_splash` 延长启动屏，掩盖初始化时间

#### 6. [Performance]: 看板渲染<500ms目标需要数据量约束

- Location: Phase 4 / P4-05 / 性能优化
- Rationale:
  - 月度看板包含: 收支卡片(1次聚合查询) + 饼图(1次分组查询) + 柱状图(1次分组查询) + 明细表(分页查询)
  - 如果月度账单超过100条，饼图和柱状图的数据处理+渲染可能超过500ms
  - fl_chart的PieChart在数据项超过15个时渲染性能下降明显

- Suggestion:
  - 明确看板渲染<500ms的前提条件: 月度账单<200条
  - 饼图数据项超过10个时，合并小项为"其他"
  - 使用 `drift` 的 `watch` 实现数据缓存，避免每次进入页面都重新查询
  - 考虑使用 `compute()` isolate处理聚合计算

#### 7. [C-Drive]: 300MB目标过于乐观，实际可能达到500-800MB

- Location: Phase 0 / 目标
- Rationale: 即使将所有工具链安装到D盘，以下C盘路径仍会产生不可控的写入:
  - `C:\Users\{user}\AppData\Local\Android\` -- Android SDK的avd和临时文件(即使SDK在D盘)
  - `C:\Users\{user}\.android\` -- Android调试桥缓存
  - `C:\Users\{user}\AppData\Local\Google\Chrome\User Data\Default\` -- Flutter Web调试时Chrome缓存
  - `C:\Users\{user}\AppData\Local\Pub\Cache\` -- 即使设置了PUB_CACHE，部分全局包仍可能写入默认位置
  - Gradle Daemon的JVM堆内存会在C盘创建临时文件
  - Windows构建工具(MSVC)的缓存默认在C盘

- Suggestion:
  - 将C盘目标修改为: 核心工具链增量<300MB，允许系统缓存额外200-500MB
  - 在P0-05中增加: 检查并清理 `C:\Users\{user}\AppData\Local\Android\` 和 `C:\Users\{user}\.android\`
  - 考虑使用符号链接(junction)将顽固的C盘缓存目录指向D盘

#### 8. [Migration]: 数据迁移工具缺少增量同步和回滚机制

- Location: Phase 2 / P2-05 / MigrationService
- Rationale:
  - 当前设计只有"全量扫描+导入"，如果导入中途失败(如解析异常)，已导入的数据如何处理?
  - 用户可能多次运行迁移，需要幂等性保证(不重复导入)
  - 导入后如果发现数据有误，缺少回滚机制

- Suggestion:
  - 增加增量迁移: 基于文件修改时间戳，仅导入新增/变更的文件
  - 增加事务性导入: 每个模块的导入在独立事务中执行，失败可回滚
  - 增加导入日志表: 记录每个文件的导入状态(成功/失败/跳过/重复)
  - 增加"清除已导入数据"功能，支持回滚

## 总体评估

**整体可行性: 有条件可行**

计划在架构设计、数据源分析、业务逻辑迁移方面做得很好，但在以下3个方面存在必须解决的技术障碍:

1. **数据库Schema不完整** -- 4个AI模块无对应表，这是数据完整性的硬伤
2. **Android 11+文件访问** -- Syncthing双通道方案的假设不成立，需要替代方案
3. **Android 14+精确通知** -- 需要额外权限和用户引导，影响通知可靠性

建议在Phase 0增加一个"技术验证Spike"，专门验证上述3个障碍的可行性，避免在Phase 3才发现方案不可行导致大量返工。
