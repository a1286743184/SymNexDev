# Ruflo 项目探索记录

> 探索时间：2026-03-24
> 背景：在现有项目中引入 AI Agent 编排工具，评估其与现有 superpower 工作流的整合可能性

---

## 一、探索动机

在开发一个已有半年的项目时，希望引入 AI 多智能体协作能力来提升开发效率。听闻 Ruflo（原 Claude Flow）是企业级 AI Agent 编排平台，支持 60+ 专业化 Agent、容错共识、自学习记忆等功能，于是开始探索。

---

## 二、探索过程

### 2.1 初步了解项目

**信息来源**：GitHub 仓库 `ruvnet/ruflo`，NPM 包 `ruflo`

快速获取了项目基本信息：
- **项目名**：RuFlo v3.5.42，原名 Claude Flow
- **定位**：企业级 AI 多智能体编排平台，将 Claude Code 扩展为多 Agent 开发环境
- **核心能力**：60+ 专业化 Agent、swarm 协作、自学习、RuVector 向量记忆、MCP 集成
- **安装**：`npm install -g ruflo`
- **License**：MIT

### 2.2 安装与 MCP 配置

尝试全局安装并注册为 Claude Code MCP Server：

```bash
# 全局安装
npm install -g ruflo

# 添加为 MCP Server（当前添加到了项目本地配置）
claude mcp add ruflo -- npx -y ruflo@latest mcp start
```

**遇到的问题**：MCP Server 健康检查显示 "Failed to connect"，但进一步诊断发现 `ruflo mcp start` 本身正常运行，stdio 模式的健康检查超时是正常的。ruflo MCP **已成功添加并可用**。

### 2.3 理解 init 的作用域

**问题**：`ruflo init` 是全局一次性初始化，还是每个项目需要单独初始化？

通过源码和文档调研确认：

- `ruflo init` 是**每个项目分别需要运行一次**的
- 它会在当前项目目录下创建本地配置文件（CLAUDE.md、hooks、agents 等）
- 全局安装的只是 CLI 本身和 MCP Server 连接
- 对于已有项目，使用 `ruflo init upgrade --add-missing` 可保留现有数据

### 2.4 理解记忆架构

**核心问题**：在 A 项目积累的开发经验，在 B 项目中是否还能用？

通过调研了解到 RuFlo 的 3 层记忆架构：

| 层级 | 范围 | 跨项目共享 |
|------|------|-----------|
| **project** | 仅当前项目 | ❌ 隔离 |
| **local** | 本地机器 | ⚠️ 有限 |
| **user** | 用户级别 | ✅ 可以跨项目 |

关键洞察：
- **项目特定知识**（如项目架构、业务逻辑）→ 存在 project 级，隔离
- **个人使用习惯和通用模式**（如调试策略、代码风格偏好）→ 存在 user 级，跨项目

### 2.5 理解记忆的触发方式

**核心问题**：经验沉淀是自动的还是需要触发？如何触发？

通过源码分析发现：

- `recordInsight()` 是一个**程序化 API**，不会自动调用
- 需要显式触发：直接调用命令、或告诉 Agent 帮你存
- 三种触发方式：
  1. **命令**：`ruflo memory_store --insight "..." --category debugging`
  2. **自然语言**：让 Agent "总结经验并保存"，Agent 可以调用 `memory_store`
  3. **Hook 配置**：在特定事件上挂载 hook 自动触发（需提前配置）

### 2.6 检查 memory tools 可用性

通过 MCP JSON-RPC 直接调用 `tools/list` 验证，ruflo 暴露了大量工具，部分与记忆相关：

- `memory_store` / `memory_retrieve` / `memory_search` — 向量记忆存储与搜索
- `hooks_post_task`（含 `storeDecisions` 选项）— 任务完成后自动存路由决策
- `agentdb_pattern_store` — 存入 ReasoningBank 向量库
- `hooks_post_edit` — 记录编辑结果用于学习

**结论**：Agent **可以直接调用** memory 工具，不需要手动操作。

### 2.7 理解 ruflo 在工作流中的位置

**这是最关键的认知转折点。**

通过深入分析 superpowers 技能和 ruflo 的 agent 架构，终于理解了两者的关系：

```
┌─────────────────────────────────────────────────────┐
│  Superpower Skills（主会话流程层）                    │
│  brainstorm → writing-plans → executing-plans →       │
│  verification-before-completion                      │
│  角色：驱动整个开发流程，决定做什么、按什么顺序        │
└──────────────────────┬──────────────────────────────┘
                       │ 主观能动调用
                       ▼
┌─────────────────────────────────────────────────────┐
│  Ruflo Agent System（执行层强化）                    │
│  Swarm 协作、Agent 派发、向量记忆、HOOKS             │
│  角色：在计划框架内执行任务，是被动的劳动者           │
└─────────────────────────────────────────────────────┘
```

**关键认知**：
- ruflo 的 Agent 是**被动的执行者**，需要你告诉它们做什么
- ruflo **不会自动**按 superpower 流程推进（它不知道你的 brainstorming、writing-plans）
- superpowers 是**流程编排层**，ruflo 是**执行层强化**

### 2.8 已有项目中使用 ruflo 的推荐流程

对于半开发状态的项目引入 ruflo，推荐步骤：

```bash
# 1. 初始化（保留现有数据）
npx ruflo@latest init upgrade --add-missing

# 2. 让 ruflo 学习项目上下文
ruflo hooks pretrain --path . --depth medium

# 3. 开发新功能时
# - 用 superpower 流程确定做什么（brainstorm → writing-plans）
# - 在 executing-plans 中调用 ruflo 执行
memory_search  # 查询相关经验（可选）
swarm_init     # 启动多 Agent 协作（可选）
# → Agent 执行
memory_store   # 让 Agent 自动或手动保存经验
```

## 三、核心结论

### 3.1 ruflo 是什么

ruflo 是一个**执行层的 AI Agent 编排平台**，不是顶层的工作流管理系统。它提供了：
- 多 Agent 协作框架（swarm、hive-mind）
- 向量记忆能力（RuVector、HNSW）
- 自学习系统（SONA、LearningBridge）
- 丰富的 MCP 工具（80+ tools）

### 3.2 与 superpower 工作流的关系

**互补关系，不是替代关系**：

| Superpower 技能 | ruflo 能做吗 |
|----------------|-------------|
| brainstorm（意图→设计） | ❌ 需你主动发起 |
| writing-plans（制定计划） | ⚠️ 可辅助，流程仍需你驱动 |
| executing-plans（执行计划） | ✅ 可派发 agent 执行 |
| verification（验证完成） | ✅ 可辅助 |
| 记忆沉淀 | ✅ `memory_store` 可用 |

### 3.3 记忆沉淀机制

- **触发**：需要显式调用 `recordInsight()` / `memory_store`
- **自然语言触发**：告诉 Agent "总结并存记忆"，Agent 可自动调用
- **自动辅助**：`hooks_post_task` 的 `storeDecisions` 可在任务结束时自动存储路由决策
- **跨项目**：user 级记忆自动跨项目；project 级记忆隔离
