# SymNex 需求澄清与架构决策

---

## 一、背景与定位

### 1.1 为什么需要开发这款软件

**原系统痛点**：

SymbiNexus 基于 Obsidian 库和自制插件体系实现了一套个人全能记录体系，经过半年以上的实际使用，验证了核心功能的价值，同时也暴露了明显的问题：

| 痛点 | 描述 |
|------|------|
| **性能问题** | Obsidian 冷启动慢（4-6秒），移动端体验差 |
| **系统集成不足** | 无法发送系统通知提醒，无法作为独立应用存在 |
| **移动端限制** | Obsidian 移动端仅作为查看工具，录入效率低 |
| **架构臃肿** | 插件+模板+DataviewJS 复杂度高，维护困难 |

**核心需求**：
- 一款独立的移动端应用，具备 Obsidian 中实现的智能录入能力
- 极快的冷启动速度（<1秒）
- 完整的系统通知集成
- 保留与 Obsidian 的数据互通能力

### 1.2 新软件与 Obsidian 的分工

**不是完全摒弃，而是重新分工**：

| 场景 | 手机端（SymNex） | 电脑端（Obsidian） |
|------|-----------------|-------------------|
| **录入** | 完整实现所有模块的智能录入 | 仅查看/编辑 md 文件 |
| **查看** | 看板、图表、统计 | md 文件浏览 |
| **知识沉淀** | - | md 格式存储 + AI 分析 |
| **数据格式** | SQLite（内部）+ md 导出 | md 文件 |
| **同步方式** | md 压缩包导出/导入 | Syncthing（未来可选） |

**关键洞察**：
- **手机是入口**：所有录入和快速查询都通过手机端 SymNex
- **电脑是沉淀**：知识整理、AI 分析、长文本编辑在 Obsidian 完成
- **md 是桥梁**：通用格式保证两端数据互通
- **不追求替代**：新软件与 Obsidian 各司其职

### 1.3 设备使用习惯

**手机为主（90%以上）**

手机端功能定位：
- 智能录入（核心）
- 看板查看
- 快速检索
- 提醒通知

电脑端用途：
- md 格式数据导出与处理
- 复杂知识整理
- Obsidian 端知识沉淀
- AI 分析处理

## 二、核心使用场景

### 2.1 最高频功能

**智能录入是唯一的核心入口**，必须覆盖所有板块功能场景，否则"单入口智能化"的特性不成立。

录入场景包括：
- 账务收支
- 任务事项（含 DEV[代码开发]/DEMO[快捷备忘] 等自定义标签）
- 美食收藏
- 价格追踪
- 杂项记录（兜底）

### 2.2 数据查看频率

**经常查看**，需要看板、图表、统计来了解整体情况。

### 2.3 性能需求

**冷启动速度是核心关注点**

- 录入响应速度也需要快（目标<1秒）
- 列表加载速度可接受正常过程
- 软件入口必须快

## 三、AI 智能录入

### 3.1 现有提示词是核心资产

原有Obsidian版实现的分类提示词经过半年实战验证，分类与字段提取都非常符合需求。

- 分类提示词应当原样保留（或仅改变不适配部分）
- 字段提取细节需要适配新数据结构
- 多供应商支持：longcat一个渠道已完全够用

### 3.2 AI 辅助板块新增（核心支柱）

**这是支撑"单入口智能录入"的核心支柱之一。**

随着用户使用，软件会越来越符合用户自己的需求——用户不仅是被动使用者，更是系统的共同塑造者。

**功能设计**：
- 用户描述需要什么板块（例如："我想记录我阅读的书籍信息，包括书名、作者、阅读状态、笔记"）
- 系统调用具备编程能力的AI模型
- AI分析用户需求 + 参考现有所有板块的提示词设计
- 自动生成新板块的分类提示词、字段提取提示词
- AI推荐录入表单结构和展示界面配置
- 用户微调确认后，新板块即可投入使用

**录后校正机制（反馈闭环）**：

1. **即时校正入口**
   - 每次智能录入完成后，弹出持续3秒的确认窗口
   - 窗口显示："已识别并录入 [某某板块]"，用户可点击确认或点击"调整"
   - 无交互则自动确认录入

2. **调整流程**
   - 用户点击"调整"后，选择目标板块
   - 将用户的原始输入内容发送给目标板块的处理管道
   - 按目标板块的标准完成录入

3. **偏差记录**
   - 记录内容：原始输入文本 + AI错误分类 + 用户预期分类
   - 作为后续提示词优化的数据源
   - 支持用户查看和管理偏差记录

4. **提示词优化入口**
   - 专门的"分类优化"功能
   - 调用已记录的偏差分类记录
   - 联合现有模块提示词上传大模型
   - 要求大模型分析偏差原因并优化提示词
   - 用户确认后应用优化
   - **优化可撤回**：保留历史版本，支持回滚

**技术要点**：

1. **分类边界保证**
   - AI需要分析现有所有板块的分类提示词
   - 确保新板块与现有板块分类界限清晰，避免分类冲突
   - 提供分类冲突检测和调整建议

2. **半固定Schema设计**
   - 基础字段固定：id/title/date/tags/content/source/folder_path/created_at/updated_at
   - AI生成扩展字段定义（字段名、类型、是否必填、验证规则）
   - 扩展字段存储于独立的ext_fields表或JSON字段

3. **动态表单生成**
   - 根据AI生成的字段定义动态渲染录入表单
   - 支持字段类型：文本/数字/日期/标签/开关/下拉/多选/富文本等
   - 支持字段验证规则（正则、范围、必填等）
   - 用户可在AI生成后微调表单结构

4. **动态展示界面**
   - 根据AI生成的展示配置渲染列表页和详情页
   - 支持自定义显示哪些字段、排序规则、筛选条件
   - 列表页支持按扩展字段筛选和排序
   - 详情页支持富文本渲染和附件展示

5. **提示词一致性**
   - AI参考现有板块的提示词设计模式
   - 新板块提示词与现有体系保持一致风格
   - 支持用户后续手动调整提示词

### 3.3 双模型配置

| 用途 | 模型要求 | 推荐 |
|------|---------|------|
| **录入功能大模型** | 便捷、准确、响应快 | longcat（现有） |
| **板块创建大模型** | 理解力强、具备编程能力 | Minimax/GLM等 |

## 六、个人信息与知识系统

### 6.1 个人数字复制体的构成

| 类型 | 内容 | 录入方式 |
|------|------|---------|
| 任务系统 | 所有做过、想做的事 | 手机智能录入 |
| 财务系统 | 消费情况 | 手机智能录入 |
| 个人信息 | 身份信息、账户密码、想吃的店、当日心得、心情等 | 手机录入（这部分应当由AI辅助板块自定义功能跑通后都可变为智能录入） |
| 个人知识库 | 知识点、错题分析、搜集资料 | **不适合手机端录入** |

### 6.2 知识库的定位

- 手机端不适合作为复杂知识的录入入口
- 手机端作为**单一查询入口**价值巨大
- 知识库在电脑端沉淀（或许是用户在Obsidian端以md格式沉淀后，可以压缩包形式手机端打包导入，转化为我们系统可用的数据结构和存储形式，如此才能做到知识库的闭环和全信息的单一入口查询全覆盖）

**关键变更说明**：
- 导出保持Obsidian式层级结构（文件夹+md文档）
- 导入解析压缩包，保留folder_path层级信息
- 所有表都包含folder_path字段，用于导航和导出重建

### 8.2 决策理由

1. **性能最优**：毫秒级查询，冷启动<500ms
2. **架构简单**：单一SQLite数据源，无双写复杂度
3. **开发聚焦**：专注于手机端功能，5个核心模块
4. **知识闭环**：md导入机制实现知识库与录入的完整闭环
5. **可演进**：需要时可在当前基础上叠加Syncthing

### 8.3 项目决策

**新开项目 + 复制复用当前有价值代码**

#### 保留部分
- AI管道3阶段 + 8供应商配置
- PivotService（枢轴偏移）
- ChannelNormalizer（渠道归一化）
- 各模块的提取提示词（适配新数据结构）

#### 数据库Schema（精简为5个核心模块 + 2个扩展模块 + 3个系统表）
| 模块 | 表名 | 说明 | 层级支持 |
|------|------|------|---------|---------|
| 账务 | bills | 收支记录 | folder_path | deleted_at |
| 任务 | tasks | 统一任务（含标签字段） | folder_path | deleted_at |
| 美食 | food_wishlist | 美食收藏 | folder_path | deleted_at |
| 价格 | price_records | 价格追踪 | folder_path | deleted_at |
| 杂项 | others | 兜底记录 | folder_path | deleted_at |
| 复盘 | reviews | 日/周/月复盘 | folder_path | deleted_at |

**软删除策略**：所有数据表采用软删除，`deleted_at` 为null表示正常记录，有值表示已删除。

#### 移除部分
- Markdown真相源设计

- SyncBridge/FileWatcher
- 原有11模块中的8个（contact/question_entry/code_dev/pet_growth/study_record/memos）

#### 新增部分
- 标签系统（统一任务下的多标签支持）

## 十、设计原则

1. **单一入口**：无论什么类型的信息，都从一个地方录入
2. **专业输出**：查看、编辑、导出都同样便捷
3. **知识闭环**：手机录入 -> md导出 -> 电脑沉淀 -> md压缩包导入 -> 系统查询
4. **AI持续进化**：用户是系统的共同塑造者，AI辅助板块新增使系统随使用而进化
5. **性能优先**：冷启动<1秒，录入响应<1秒
6. **本地为主**：数据本地SQLite存储，md可选导出/导入
7. **标签化泛化**：DEV/DEMO等通过标签区分，统一任务管理
8. **层级化存储**：所有数据保留层级结构，支持按文件夹导航

## 十一、数据导入导出设计

### 11.1 核心原则

- **内部存储**：所有数据以SQLite为唯一数据源
- **导出层级化**：导出不是散落的md文档，而是保持Obsidian式的层级结构
- **导入结构化**：Obsidian的文件夹层级结构是重要的元信息，导入时必须保留并转化

### 11.2 md导出设计

导出时保持我们自己的层级结构（而非复制Obsidian原结构）：

```
导出压缩包/
├── 账务/
│   ├── 2026/
│   │   ├── 01/
│   │   │   ├── 260101-01-餐饮消费-堂食.md
│   │   │   └── 260102-01-交通出行-停车.md
│   │   └── 02/
│   │       └── ...
│   └── ...
├── 任务/
│   ├── DEV/
│   │   ├── 完成/
│   │   │   └── xxx.md
│   │   └── 待办/
│   │       └── xxx.md
│   ├── DEMO/
│   │   └── ...
│   └── ...
├── 美食/
│   └── ...
├── 价格追踪/
│   └── ...
├── 复盘/
│   ├── 日复盘/
│   │   └── 2026-01-01.md
│   └── 周复盘/
│       └── 2026-W01.md
├── 知识库/
│   ├── 个人资料/
│   │   └── 证件信息.md
│   └── 学习笔记/
│       └── 行测错题集.md
└── 其他/
    └── ...
```

**导出粒度**：
- 单条记录导出：某一条账单、某一个任务
- 板块导出：整个"账务"模块导出
- 整体导出：所有数据，按层级结构打包

**导出格式**：
- 每个md文件保持Obsidian兼容的frontmatter
- 层级结构通过文件夹体现
- 文件命名遵循我们自己的规范

### 11.3 md导入设计

导入时解析Obsidian导出的压缩包，保持层级结构映射：

**导入流程**：
1. 用户选择md压缩包（或指定文件夹路径）
2. 解析压缩包中的文件夹结构
3. 遍历所有md文件，提取frontmatter
4. 根据frontmatter字段识别模块类型
5. 根据文件路径层级建立内部关联
6. 写入SQLite数据库

**模块识别规则**：
| 识别条件 | 目标表 |
|---------|-------|
| frontmatter `type: bill` 或位于 `账务/` 文件夹下 | bills |
| frontmatter `type: task` 或 checkbox格式 或位于 `任务/` 文件夹下 | tasks |
| frontmatter `type: food_wishlist` 或位于 `美食/` 文件夹下 | food_wishlist |
| frontmatter `type: price_tracker` 或位于 `价格追踪/` 文件夹下 | price_records |
| frontmatter `type: review` 或位于 `复盘/` 文件夹下 | reviews |
| frontmatter `type: knowledge` 或位于 `知识库/` 文件夹下 | knowledge |
| 其他情况 | others |

**路径匹配规则**：
- 顶层文件夹名称作为模块标识：`账务/`、`任务/`、`美食/`、`价格追踪/`、`复盘/`、`知识库/`
- 匹配时优先检查frontmatter的type字段，若无则检查文件路径
- 支持多级嵌套：`账务/2026/01/` 等

**层级结构映射**：
- 文件夹路径作为模块分类和导航的基础
- 支持多级嵌套
- 支持标签提取（从文件名或frontmatter的tags字段）

## 十二、用户界面与交互设计

### 12.1 产品定位分析

**核心定位**：效率型个人生活管理应用

| 维度 | 分析 | 设计决策 |
|------|------|---------|
| **使用场景** | 碎片化录入、快速检索、定期回顾 | 入口浅、反馈快、信息密度高 |
| **核心用户** | 个人用户，移动端为主(90%) | 移动端优先，触摸友好 |
| **设计目标** | 录入效率最大化 | 最小化录入摩擦 |
| **差异化** | AI智能分类、持续进化 | 智能化反馈机制 |

### 12.2 设计原则

1. **入口即录入**：首页即录入界面，核心功能零层级触达
2. **所见即操作**：列表项可直接操作，减少跳转
3. **反馈即时化**：每个操作都有明确反馈
4. **信息分层**：基础信息可见，高级信息可展开
5. **容错设计**：误操作可撤回，分类错误可校正

### 12.8 状态反馈设计

- 列表加载：骨架屏占位
- AI处理中：进度条 + 状态文字
- 页面切换：内容渐入

**空状态**：
- 首次使用：引导卡片 + 操作指引
- 无数据：友好提示 + 快捷入口
